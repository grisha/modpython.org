</div>PHP is hardly the indicator of good practice.<br>
<br>
In web applications I would regard it as bad practice to hard wire<br>
into configuration files URLs for the same site which incorporate the<br>
name of the site, which seems to be what you are doing here.<br>
<br>
The configuration file can contain an absolute path to a page, but it<br>
should not be a fully formed HTTP address. Any code generating a<br>
response which requires a fully formed HTTP address should take that<br>
absolute path and feed it into req.construct_url(). This maps to an<br>
internal Apache function which will fill it out to be a fully formed<br>
HTTP address with all the correct hostname and port information as<br>
matches what the original request to the site used.<br>
<br>
In other words, if you do things the correct way of using<br>
req.construct_url() you don&#39;t need to hardwire your hostname into your<br>
global configuration files.<br>
<br>
If for some reason the code generating the response doesn&#39;t have<br>
access to the request object, then you should be looking at Python<br>
features such as thread locals to stash away some request information<br>
that can then be accessed by code directly which otherwise didn&#39;t get<br>
passed the request object. This would allow it then to get hold of the<br>
request object and call the construct_url() function.<br>
<br>
That all said, why do you even need a fully formed HTTP address? If<br>
you are including that in pages for the same site then that also could<br>
be considered bad practice. You should just use an absolute, or better<br>
still a relative path, in any href attributes of a HTML anchor tag,<br>
not a full formed HTTP address.<br>
