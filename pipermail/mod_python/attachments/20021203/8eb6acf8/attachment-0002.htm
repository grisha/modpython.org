<P>The new dll doesn't solve the problem.&nbsp; However, putting the msvcr70.dll into the system32 directory does solve the problem. Thanks to Duncan Booth for the solution:</P>
<P>&gt;It appears that the binaries for mod_python 3.0.1 have been built <BR>&gt;using Microsoft Visual Studio version 7. This means that <BR>&gt;mod_python.so has a dependency on the file msvcr70.dll, but the <BR>&gt;distribution doesn't include this file.<BR><BR>&gt;If you have the latest Microsoft compiler installed, or any other <BR>&gt;applications that use this DLL, then mod_python.so works fine. If not <BR>&gt;it fails horribly.<BR><BR>&gt;The workaround (at least on one machine that had this problem) is to <BR>&gt;copy msvcr70.dll into \winnt\system32. Get the file from anywhere <BR>&gt;that already has a copy (e.g. www.dll-files.com/dllindex/dll-<BR>&gt;files.shtml?msvcr70). Fortunately this dll has no further <BR>&gt;dependencies.&nbsp; <BR><BR>&gt;The long term solution would be to distribute mod_python as a .MSI <BR>&gt;file including dependencies, or even to build it with the same version <BR>&gt;of the C runtime library as used for the rest of Apache and Python.<BR><BR><FONT color=#0000ff>Chad Harrington </FONT></P></DIV><FONT color=#0000ff>harrington_chad@hotmail.com</FONT>
